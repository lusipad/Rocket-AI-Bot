# Rocket.Chat Team Copilot 产品方向文档

状态：Draft

最后更新：2026-04-26

负责人：待定

## 1. 一句话定义

Rocket AI Bot 的下一阶段目标，不是成为“会聊天的工具集合”，而是成为 Rocket.Chat 里的团队协作 AI——从代码和工单入手，逐步覆盖多人讨论、资料补齐、结论沉淀和动作闭环。

## 2. 背景判断

当前项目已经具备可用基础：

- Rocket.Chat 群聊与私聊对话
- 多轮上下文
- 图片理解
- 模型原生联网搜索
- 本地仓库检索（ripgrep + 文件读取）
- Codex CLI 代码分析
- Azure DevOps 查询
- 定时任务调度
- Web Admin 管理端

这说明项目已经跨过 demo 阶段。

但当前产品形态仍然偏“问一句，答一句”的 bot。它更像聊天入口上的工具编排层，还不像真正嵌入团队协作流程的工作型 AI。

核心问题不在于“还能再接什么工具”，而在于：

1. bot 是否能在具体工作场景中给出有来源、可验证的回答
2. bot 是否能接住一段真实讨论，而不是只看当前一句话
3. bot 的输出是否能继续流转成待办、工单、提醒、播报，而不是停在聊天窗口里

## 3. 产品结论

产品应当被定义为：

**Rocket.Chat 里的 Team Copilot，从 DevTools 场景切入，逐步覆盖团队协作全流程。**

这个定义意味着四件事：

1. 第一步不是跳进模糊的“讨论理解”，而是先在代码和工单这类结构化场景中打磨工具编排、来源引用和输出质量
2. 在 DevTools 能力成熟后，自然延伸到“讨论总结 + 资料补全”的协作场景
3. 核心价值是减少信息整理和结果流失，不是展示模型能力
4. 竞争力来自私有部署、内部系统接入、权限边界和低打扰协作体验

## 4. 目标用户

优先用户：

- 研发团队
- 运维 / SRE 团队
- 技术项目负责人
- 内部支持与交付团队

这些用户有几个共同点：

- 长时间在 Rocket.Chat 里讨论技术问题
- 讨论中需要频繁查代码、查工单、补资料
- 讨论结果经常需要落到代码、工单、发布、排障、日报上
- 对数据边界、内部系统接入和可审计性敏感

## 5. 核心使用场景

### 场景 A：代码与工单查询（Phase 1 重点）

研发人员在频道里 @ bot，希望它：

- 搜索本地仓库代码，定位函数定义、调用链、配置项
- 读取关键文件，解释实现逻辑
- 查询 Azure DevOps 工作项状态、PR 详情、Pipeline 运行结果
- 将多个工具的结果组合成一个有来源的完整回答

这个场景是产品从“能聊天”升级为“能干活”的第一步，也是技术基础最成熟的切入点。

### 场景 B：PR 审查辅助（Phase 1 重点）

用户将 bot 拉到 PR 讨论中，希望它：

- 根据 PR 描述查找关联代码
- 分析变更影响面
- 调用 Codex 做代码质量分析
- 结合 Azure DevOps 工单上下文给出审查建议
- 输出附代码引用和工单来源

### 场景 C：Pipeline 监控播报（Phase 1 重点）

团队需要固定频率获取：

- Pipeline 编译 / 测试 / 部署状态
- PR 状态汇总
- 工作项燃尽风险
- 异常主动告警

这个场景当前 scheduler 已有基础，需要做的是产品化为模板任务和结构化播报。

### 场景 D：多人讨论后拉 bot 进来接管（Phase 2 重点）

用户在群里讨论了多轮后，把 bot 拉进来，希望它：

- 总结目前结论
- 提炼分歧点
- 列出待办和 owner
- 指出还缺哪些信息
- 补充联网资料、代码证据或 Azure DevOps 状态

### 场景 E：讨论中即时补资料（Phase 2 重点）

团队正在讨论一个问题，bot 被调用去：

- 联网搜索最新公开信息
- 查询本地仓库代码
- 读取关键配置或实现文件
- 查询 Azure DevOps 工作项、PR、流水线状态

bot 的价值不是“我能搜”，而是“我能在这段讨论的上下文里补证据”。

### 场景 F：讨论结果转执行（Phase 3 重点）

讨论完成后，bot 不应该只输出一段文字，还应支持：

- 生成结构化待办
- 创建 Azure DevOps work item
- 把总结回发线程或频道
- 发送提醒或后续播报

### 场景 G：知识沉淀与复盘（Phase 4 重点）

长期运行后，bot 积累的讨论和决策应可被检索和复用：

- 历史决策追溯——“为什么当时选了方案 A”
- 故障时间线重建——汇总聊天记录 + PR 合并 + 部署记录
- 新人上手——“这个项目的认证逻辑在哪”

## 6. 不做什么

在对应阶段成熟之前，明确不优先做以下方向：

- 不做通用聊天机器人
- 不做长期全量公共频道记忆
- 不以接更多工具为第一目标
- 不优先做复杂高风险自动执行（Phase 3 之前）
- 不先做宽而浅的“万能企业 copilot”

产品必须先把少数高频场景打穿，再扩能力面。

## 7. 产品原则

### 7.1 结果优先，不是模型优先

用户要解决的是查代码、审 PR、补信息、沉淀结论和推进任务，不是单纯体验大模型。

### 7.2 协作优先，不是回答优先

理想输出不是所有场景都写成固定模板。

更合理的原则是：

- 默认像同事一样自然回复
- 当用户是在“拿结果”，而不是“聊一下”时，再切到制品输出

只有在制品场景里，才更适合输出：

- 结论
- 证据（代码引用、工单链接、联网来源）
- 分歧
- 待办
- owner
- 风险
- 下一步动作

### 7.3 低打扰优先

公共频道里，聪明很重要，但不打扰更重要。

默认触发方式应该保持克制：

- 被 `@`
- 私聊
- 明确命令
- 定时任务
- 可选的长讨论后建议总结

### 7.4 来源与审计是卖点，不是附加项

只要回答涉及联网、内部检索、工单或执行动作，就应尽量做到：

- 有来源
- 可追踪
- 可解释
- 可审计

但来源展示不应一刀切：

- 普通查询，轻量引用即可
- 制品输出，再显式列出来源

### 7.5 有限记忆，而不是无限记忆

长寿公共频道不适合做全量喂给模型。

正确做法应当是三层结构：

1. 当前讨论窗口
2. 滚动摘要快照
3. 按需回溯更早消息

### 7.6 从确定性高的场景切入

不从最模糊、最难评估的“讨论理解”开始，而是从代码和工单这类结构化场景开始——

- 先把工具编排、来源引用、输出质量打磨到位
- 再把能力迁移到“多人讨论”场景
- 每一步都有明确的验收标准

## 8. 北极星指标

北极星不是总调用量，而是团队协作里真实被依赖的程度。

分阶段重点观察：

**Phase 1：**
- 每周活跃房间数
- 代码 / 工单查询次数
- 回答带来源比例
- PR 审查被采纳率

**Phase 2-4（叠加）：**
- 每周“会话接管”次数
- 总结后被继续引用或追问的比例
- 回答后触发动作比例
- 负反馈率
- 知识检索命中率

## 9. 路线图

### 总览

```
Phase 1           Phase 2            Phase 3              Phase 4
DevTools Copilot  → 会话接管         → 结果对象化         → 知识沉淀
代码+工单+PR      讨论总结+补资料     待办+工单+播报模板   审计+策略+复盘
─────────────────────────────────────────────────────────────────────
地基：来源引用、工具编排、制品类结构化输出
```

四阶段的核心逻辑：

- **Phase 1 不是捷径，是地基。** DevTools Copilot 把来源引用、工具编排、制品类结构化输出打磨好——这三样恰好是后续所有能力的前提。
- **Phase 2 不再跳跃。** 当 P1 已经能让 bot 在代码和工单场景里给出有来源的回答时，“讨论总结”只需要把输入从结构化数据扩展到人类对话，引擎是同一套。
- **Phase 3-4 是自然延伸。** 不设“远景目标”，每一阶段的交付物在独立可用的前提下累加新能力。

---

### Phase 1：DevTools Copilot（建议 4-6 周）

**目标：** 让 bot 在代码、工单和流水线场景中成为可依赖的工具，同时完成制品类输出和来源引用的基础设施建设。

**当前基础：** `search_code` + `read_file` + `azure_devops` + `exec_codex` + `scheduler` 已在生产运行。代码库匹配度约 70%。

**优先交付：**

- 来源引用基础设施
  - `ToolResult` 接口统一 `sources` 字段（文件路径 + 行号、工单链接、PR 链接）
  - 系统提示词按场景决定是否显式展示来源
  - 来源格式标准化为文件路径:行号（如 `src/bot/client.ts:42`）
- PR 审查能力
  - 根据 PR 编号查询详情 + 关联代码变更
  - Codex 代码分析集成
  - 审查建议附代码引用和工单上下文
- Pipeline / 工单监控播报
  - 预置模板任务：PR 状态汇总、Pipeline 健康检查、工作项燃尽
  - 结构化播报输出（替代“一段文本”）
  - 异常主动推送
- 度量基础设施
  - 请求类型分类（代码查询 / 工单查询 / PR 审查 / 播报）
  - 来源引用率统计
  - 房间活跃度统计

**验收标准：**
- 代码 / 工单查询类回答中，≥70% 附文件路径:行号或工单链接
- PR 审查场景下，bot 能关联代码变更 + Azure DevOps 上下文
- 至少 3 个 Pipeline / PR 相关模板任务可用
- 管理员能从度量数据判断 bot 的使用情况

**对当前代码的主要改动：**
- `tools/` 各工具返回值增加 `sources` 字段
- `agent/orchestrator.ts` 系统提示词重写，面向 DevTools 场景
- `agent/orchestrator.ts` `recentMessages.slice(-20)` → 可配置的消息数量
- `web/` 增加基础度量 API（请求统计、来源引用率）
- `scheduler/` 增加模板任务定义

---

### Phase 2：会话接管（建议 6-8 周）

**目标：** 让 bot 真正接住多人讨论——总结结论、提炼分歧、补充资料。

**前置依赖：** Phase 1 的来源引用、工具编排和度量能力。

**优先交付：**

- `SessionManager` 模块
  - 房间 / 线程级滚动摘要快照（生成、存储、更新）
  - 三层记忆结构落地：当前窗口 → 摘要快照 → 按需回溯
  - 关键决策和待办自动提取
- 上下文组装策略重写
  - 从“最近 N 条消息”改为“摘要 + 最近窗口 + 按需回溯”
  - 长讨论自动分段 + 关键信息提取
  - `room_history` 工具自动触发（不再依赖模型判断）
- 面向讨论场景的专用输出模板
  - 结论、分歧、待办、owner、风险、来源
- 联网资料 + 代码证据的上下文补全

**验收标准：**
- 在 30 到 100 条消息的多人讨论后仍能给出有用总结
- 不需要用户重新解释“刚才聊了什么”
- 涉及联网或检索的结论尽量附来源
- 摘要快照在 24 小时内保持可用

**对当前代码的主要改动：**
- 新建 `src/session/` 模块（SessionManager + 摘要持久化）
- 重写 `src/bot/message-handler.ts` 上下文获取策略
- 重写 `src/llm/context.ts` ContextBuilder，支持摘要 + 窗口 + 回溯
- `src/agent/orchestrator.ts` 增加会话接管模式的系统提示词分支

---

### Phase 3：结果对象化与动作闭环（建议 4-6 周）

**目标：** 让 bot 的输出可以继续操作——创建工单、回发线程、结构化播报。

**前置依赖：** Phase 2 的讨论理解和结构化输出能力。

**优先交付：**

- 结构化输出对象
  - `summary` —— 结论
  - `decisions` —— 已达成决策
  - `open_questions` —— 待解决问题
  - `action_items` —— 待办 + owner
  - `sources` —— 来源列表
- 一键创建 Azure DevOps work item
- 一键将总结回发当前线程或频道
- 定时任务结果的结构化历史记录（替代 JSON 文件堆）
- 常用任务模板浏览器（Admin 面板中）

**验收标准：**
- 高价值回答中，明显一部分可直接转成待办或工单
- 定时任务结果不再只是“发了一段文本”
- 任务模板可在 Admin 面板中浏览和选用

**对当前代码的主要改动：**
- `src/agent/orchestrator.ts` 增加结构化输出解析层
- `src/tools/azure-devops.ts` 增加 work item 创建能力
- `src/web/` Admin 面板增加模板浏览器
- `src/scheduler/persistence.ts` 升级为结构化历史记录

---

### Phase 4：知识沉淀与治理（建议 4-6 周）

**目标：** 让团队愿意长期打开它，让管理员能看懂它在做什么。

**前置依赖：** Phase 1-3 的积累。

**优先交付：**

- 历史决策可检索
  - 基于摘要快照的关键词搜索
  - “为什么选了方案 A”类问题的回溯能力
- 故障时间线重建
  - 汇总特定期限内的聊天摘要 + PR 合并 + 部署记录
- 治理能力
  - 高风险动作确认（创建工单、修改配置等）
  - 审计日志（谁、何时、调了什么工具、结果如何）
  - 房间级策略（是否允许自动总结、播报频率限制）
  - 用户反馈机制（thumbs up/down）
  - 失败原因可见性

**验收标准：**
- 管理员能从审计日志看懂 bot 做过什么
- 用户知道结果从哪来、为什么失败
- 公共频道不会被 bot 过度打扰
- 历史讨论中的关键决策可被检索

**对当前代码的主要改动：**
- 新建 `src/audit/` 模块（审计日志）
- 新建 `src/knowledge/` 模块（摘要索引和检索）
- `src/web/` Admin 面板增加审计日志查看、房间策略配置
- `src/bot/message-handler.ts` 增加反馈处理（reaction / 命令）

## 10. 产品与实现的对应关系

结合当前仓库，以下判断成立：

**Phase 1 基础最成熟：**
- `search_code`（ripgrep）、`read_file`、`exec_codex`、`azure_devops` 已在生产运行
- `scheduler` 已支持定时播报，缺少的是模板化和结构化输出
- 工具注册和编排的模式已经稳定，新增来源字段是增量改进

**Phase 2-4 需要新建模块：**
- `SessionManager`（会话摘要管理）—— 当前代码完全不存在
- `结构化输出层` —— 当前 orchestrator 返回纯文本
- `审计日志` —— 当前没有记录
- `知识索引` —— 当前没有

**因此，不能跳过 Phase 1 直接做 Phase 2。** Phase 1 的场景更容易验证，技术风险更低，而且交付的来源引用和结构化输出恰好是 Phase 2 的前置条件。

## 11. 决策建议

如果只能押一个方向，应该押：

**Phase 1：DevTools Copilot。**

不是因为它是最酷的方向，而是因为：

1. 当前代码 70%+ 的能力已经在这个方向上，改动最小、确定性最高
2. 它逼你做好来源引用、工具编排和结构化输出——这些是后续所有能力的地基
3. 在代码和工单场景中验证了“有来源、可追踪、能托付”之后，再去做“讨论总结”不再是一个跳跃
4. 如果 Phase 1 做不扎实，Phase 2 的“会话接管”大概率会陷入“回答看起来很聪明但不一定对”的境地

DevTools Copilot 做成后，产品就有了一个可用的、带来源引用的、被真实使用的底座。在这个底座上往 Team Copilot 方向走，每一步都有退路。
