# DevTools Copilot V1 范围文档

状态：Draft

最后更新：2026-04-26

关联文档：

- `docs/designs/rocketchat-team-copilot-prd.md`

## 1. 范围声明

本文档定义 Rocket AI Bot 的 **V1** 范围。

这里的 V1，不是整个 Team Copilot 终局，而是路线图中的 **Phase 1：DevTools Copilot**。

V1 的目标很明确：

- 先把代码、工单、PR、Pipeline 这类结构化场景做扎实
- 先把来源引用、工具编排、制品类输出打磨到可依赖
- 不在 V1 内引入会话接管、长期知识沉淀、复杂动作闭环

一句话说，V1 要解决的是：

**让团队在 Rocket.Chat 里，把 bot 当成一个可依赖的代码与工单助手。**

## 2. V1 成功定义

V1 发布后，用户至少应当愿意在这几类场景里反复使用 bot：

- 查代码定义、调用链、配置项
- 读文件并解释实现逻辑
- 查 Azure DevOps 工作项、PR、Pipeline 状态
- 对指定 PR 做审查辅助
- 订阅固定频率的 PR / Pipeline / 工单播报

V1 不追求“聪明得像人”，只追求三件事：

1. 回答有来源
2. 工具调度稳定
3. 输出能直接被人继续工作

## 3. 假设与约束

当前 V1 基于这些前提成立：

- 现有 `search_code`、`read_file`、`azure_devops`、`exec_codex`、`scheduler` 继续沿用
- 入口仍以 Rocket.Chat `@mention`、私聊、定时任务为主
- 不新增长期会话记忆系统
- 不在 V1 内创建 Azure DevOps work item
- 不在 V1 内做自动审批、审计治理、知识库索引
- 尽量不引入新的外部依赖，优先复用现有模块

如果这些前提变化，V1 范围也要一起重审。

## 4. 目标用户与核心任务

V1 优先服务这四类用户：

- 研发工程师
- 运维 / SRE
- Tech Lead / 项目负责人
- 内部支持工程师

他们在 V1 中的核心任务是：

- 快速定位代码位置和实现逻辑
- 快速知道一个 PR / Pipeline / 工单当前是什么状态
- 在聊天窗口里拿到足够可信的证据，而不是再切换多个系统
- 让固定播报任务稳定输出，不用每次重写 prompt

## 5. V1 必做范围

V1 只做 P0。没有 P0 之外的“顺手加一点”。

### 5.1 代码与工单查询

用户可以在频道或私聊里直接问：

- 某个函数 / 类 / 配置项在哪里定义
- 某段逻辑是怎么实现的
- 某个工作项、PR、Pipeline 当前状态是什么
- 某个问题是否能从代码和工单上下文中找到证据

V1 必须做到：

- 正确调用 `search_code`、`read_file`、`azure_devops`
- 回答中附来源
- 当答案依赖多个工具时，能合并成一个完整回复
- 回答默认偏简洁，不输出大段原文

### 5.2 PR 审查辅助

用户可以明确请求 bot 审查某个 PR。

V1 必须做到：

- 根据 PR 编号或上下文拿到 PR 基本信息
- 关联相关代码位置和工作项上下文
- 在需要时调用 Codex 做进一步代码分析
- 默认自然回复，不强制固定模板
- 当用户明确要求“整理成可转发/可提交的审查摘要”时，再输出固定结构

建议输出格式：

- `结论`
- `主要风险`
- `影响范围`
- `待确认问题`
- `来源`

### 5.3 定时播报模板

V1 不做开放式“随便让用户写任何任务模板平台”，先做固定高频模板。

至少提供 3 类模板：

- PR 状态汇总
- Pipeline 健康检查
- 工作项风险摘要

V1 必须做到：

- 管理端可创建和编辑这些模板任务
- 任务执行结果可追踪
- 失败原因可见
- 作为制品输出，默认使用固定结构

### 5.4 来源引用基础设施

这部分不是附属项，是 V1 的基础设施。

V1 必须做到：

- 工具返回统一的 `sources`
- 本地代码来源可定位到文件路径和行号
- Azure DevOps 来源至少包含对象类型和 ID，能附链接更好
- 联网结果若被使用，回答中应能明确说明来自联网结果
- 默认自然回复场景下，不强制每次都展示“来源”小节
- 制品场景下，显式展示来源

### 5.5 最小度量能力

V1 不做完整 BI，但必须能看清 bot 是否真的在被用。

至少记录和展示：

- 请求类型计数
- 来源引用率
- 定时任务成功率
- 活跃房间数

## 6. V1 明确不做

这些内容全部排除在 V1 之外：

- 房间 / 线程滚动摘要
- 自动会话接管
- 长期知识库 / 决策检索
- Azure DevOps 工单创建
- 审批流
- 房间级复杂策略系统
- 复杂多 Agent 工作流
- 面向全公司的通用知识问答

原则很简单：

**V1 只证明 DevTools Copilot 成立，不证明整个 Team Copilot 已成立。**

## 7. 用户可见体验

V1 对用户的可见变化，应该集中在这三类体验上。

### 7.1 默认像同事，不像报告机

普通查询默认应该：

- 先给结论
- 需要时补 1 到 2 个关键证据
- 不强制写成模板
- 不默认附“来源”小节

只有当用户明确要求生成制品，或者任务本身就是播报 / 模板输出时，才进入固定结构。

### 7.2 来源可见，但不过度展示

只要用了代码、工单、PR、Pipeline、联网信息，用户就应能看见来源。

示例：

- `src/bot/client.ts:179`
- `Work Item #1234`
- `PR #567`
- `Pipeline build-main #890`

但是否显式展示为“来源”小节，应由场景决定：

- 普通查询：按需轻量引用
- 制品输出：显式列来源

### 7.3 定时任务可预期

定时播报不是“偶尔发一段不稳定文本”，而是固定主题、固定结构、固定频率的播报。

## 8. 验收标准

V1 是否可发布，用下面的标准判断。

### 8.1 功能验收

- 代码 / 工单查询链路可用
- PR 审查辅助链路可用
- 至少 3 个定时播报模板可用
- 管理端能查看任务历史和基础状态

### 8.2 质量验收

- 代码 / 工单查询类回答中，至少 70% 带来源
- PR 审查回复中，必须包含来源
- 定时任务失败时，历史记录能看出失败原因
- 不破坏现有私聊、`@mention`、任务管理流程

### 8.3 测试验收

至少补到这些测试：

- 工具返回 `sources` 的单元测试
- orchestrator 对来源输出规则的单元测试
- PR 审查场景的编排测试
- 定时任务模板和结果记录测试
- 管理端相关接口测试

## 9. 模块映射

下面这张表定义 V1 的主要实现落点。

| 能力 | 主要文件 / 模块 | 说明 |
| --- | --- | --- |
| 工具来源统一 | `src/tools/*.ts`、`src/tools/registry.ts` | 为工具结果增加统一 `sources` 结构 |
| 回答结构与提示词 | `src/agent/orchestrator.ts` | 默认自然回复，只在制品场景启用固定结构 |
| LLM 工具编排 | `src/llm/client.ts`、`src/llm/context.ts` | 保持现有编排，补充来源相关输入约束 |
| 代码 / 工单主链路 | `src/index.ts`、`src/bot/message-handler.ts` | 保持触发入口，增强请求分类和路由 |
| PR 审查辅助 | `src/tools/azure-devops.ts`、`src/tools/codex.ts` | 以现有工具为基础拼装 PR review 流程 |
| 定时播报模板 | `src/scheduler/index.ts`、`src/scheduler/persistence.ts` | 定义模板任务和结构化结果历史 |
| 管理端展示 | `src/web/api/*.ts`、`web-admin/src/pages/*` | 展示任务、日志、最小度量 |

## 10. 开发顺序

V1 建议按这个顺序推进。

### Step 1：来源引用基础设施

先统一工具返回结构，再改提示词和来源展示策略。

原因：

- 这是 V1 最硬的基础
- 也是后续制品输出和播报模板复用最多的能力

### Step 2：代码 / 工单查询主链路

先把最常见的“查代码、查工单、查状态”打稳。

原因：

- 使用频次最高
- 验证成本最低
- 最容易快速看到来源引用是否真的有价值

### Step 3：PR 审查辅助

在来源和主链路稳定后，再加 PR review。

原因：

- PR 审查对输出结构要求更高
- 需要复用代码检索、工单查询和 Codex 分析

### Step 4：定时播报模板和最小度量

最后补模板任务和基础度量。

原因：

- 依赖前面输出结构稳定
- 能把已有 scheduler 变成真正可用的产品能力

## 11. 进入 Phase 2 的升级条件

V1 完成后，不是自动进入 Phase 2。

只有满足下面条件，才进入“会话接管”阶段：

- 代码 / 工单查询回答的来源覆盖率稳定达到目标
- PR 审查链路已经稳定可用
- 定时模板任务至少有一批真实用户在用
- 已经能从度量上判断哪些场景最值得继续深挖

如果这些条件没满足，就继续把 V1 打磨好，而不是急着进入讨论总结。

## 12. 最终判断

V1 做成的标准，不是“功能列表已经勾完”。

而是下面这句话成立：

**团队真的开始在 Rocket.Chat 里用它查代码、查工单、看 PR、看 Pipeline，而且愿意继续用。**

这才是 V1。
